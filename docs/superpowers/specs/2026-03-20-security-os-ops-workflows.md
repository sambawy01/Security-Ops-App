# Security OS for El Gouna — Ops Workflows, RBAC, Escalation & SLA Design

**Date:** 2026-03-20
**Status:** Approved
**Session:** 3 of 5

## Summary

Operational design for the Security OS MVP. Defines shift workflows, incident lifecycle with SLA enforcement, escalation chains, role-based access control for 7 system roles, and structured shift handover. Builds on the Session 2 technical architecture.

## Context

- **Personnel:** 637 total — 584 field officers, 53 management (1 security manager, 3 assistant managers, 4 secretaries, supervisors, ops room staff, manpower/admin)
- **Shifts:** 2 rotations, 12 hours each (6AM-6PM, 6PM-6AM)
- **Checkpoints:** 180 across El Gouna
- **Incident categories:** 9 defined categories
- **SLA:** 5-minute response for all priorities, graduated resolution times

## Shift Operations

### Shift Structure

| Shift | Hours | Overlap |
|-------|-------|---------|
| Day | 06:00 – 18:00 | 30 min handover window (17:30 – 18:00) |
| Night | 18:00 – 06:00 | 30 min handover window (05:30 – 06:00) |

**Officer assignment:** Each officer is assigned to a zone and shift via the scheduling system. Manpower/HR admin manages the schedule; supervisors can make same-day adjustments.

### Shift Lifecycle

```
SCHEDULED → CHECK_IN → ACTIVE → HANDOVER → CHECK_OUT → COMPLETED
                                                        ↓
                                                     NO_SHOW (if no check-in within 30 min of scheduled start)
```

1. **Scheduled:** Shift appears on officer's mobile app 24 hours before start
2. **Check-in:** Officer opens app, confirms check-in. System records GPS + timestamp. Must be within assigned zone (500m tolerance).
3. **Active:** Officer appears on command map. GPS tracking active (30s interval). Patrol routes and incident assignments enabled.
4. **Handover:** Triggered at T-30min before shift end. System generates handover brief. Outgoing officer/supervisor writes handover notes.
5. **Check-out:** Officer confirms check-out after handover acknowledged by incoming counterpart. GPS + timestamp recorded.
6. **Completed:** Shift closed. Performance metrics calculated.
7. **No-show:** If officer doesn't check in within 30 min of scheduled start, system marks NO_SHOW and alerts zone supervisor.

### Shift Exceptions

**Sick leave / call-off:**
- Officer or supervisor reports sick leave via app or phone call to ops room
- Ops room marks shift as `CALLED_OFF` with reason
- System triggers replacement suggestion: AI recommends available off-duty officer in same zone based on recent rest time and skill match
- Supervisor or HR admin approves replacement and notifies the replacement officer
- If no replacement available, supervisor is alerted to redistribute zone coverage

**Emergency overtime:**
- If incoming shift is short-staffed (2+ no-shows or call-offs), system flags to assistant manager
- Assistant manager can extend outgoing officers with one-click approval in dashboard
- Extended officers get a new shift record linked to the original, with overtime flag
- Maximum extension: 4 hours (configurable)

**Mid-shift zone reassignment:**
- Supervisor or assistant manager reassigns officer to different zone via dashboard
- System automatically: pauses current patrol (if active), transfers open incidents to zone's remaining officers or new incoming officer, updates GPS tracking to show officer in new zone
- Reassignment logged in audit_logs with reason

**Late check-in:**
- Officer checking in >30 min late: shift status remains SCHEDULED until check-in, then moves to ACTIVE
- No-show alert is sent at +30 min but can be cleared by supervisor if officer communicates delay
- System records actual vs scheduled check-in time for performance metrics

### Shift Handover Process

**Automated handover brief (generated by system at T-30min):**
- Open incidents in zone with current status and SLA countdown
- Officer positions and patrol completion status
- Notable events from the shift (escalations, anomalies, resolved critical incidents)
- AI-generated summary of shift activity (1-2 paragraphs, Arabic)

**Handover workflow:**
1. System generates brief → pushed to both outgoing and incoming supervisors
2. Outgoing supervisor adds manual notes (free text + voice note option)
3. Face-to-face briefing using the system-generated brief as agenda
4. Incoming supervisor acknowledges handover in the system (digital signature)
5. Open incidents re-assigned to incoming shift's officers (auto-suggested by smart dispatch, supervisor confirms)
6. Outgoing shift officers check out once handover is acknowledged

**If handover is not acknowledged within 30 min:** Alert escalates to assistant manager.

**Dispatch during handover window:** During the 30-minute handover window, smart dispatch prefers incoming shift officers for new incidents (scoring boost for incoming officers, penalty for outgoing). This prevents assigning incidents to officers about to check out.

### Operations Room Coverage

Operations room staff (~8-10) work the same 2x12hr shift structure as field officers. They follow the same shift lifecycle (scheduled → check-in → active → handover → check-out → completed) but:
- Check-in is at the ops room workstation (no GPS requirement)
- No patrol assignments
- Handover brief includes: open incidents across all zones, pending escalations, WhatsApp queue status, system health
- Minimum ops room coverage: 2 operators per shift at all times. If below minimum, assistant manager alerted.

## Incident Management

### Incident Categories

| # | Category | Default Priority | Typical Response |
|---|----------|-----------------|-----------------|
| 1 | Security Threat | Critical | Armed response / immediate containment |
| 2 | Fire/Safety | Critical | Evacuation coordination, fire response |
| 3 | Accidents | High | First aid, scene management, documentation |
| 4 | Trespassing | High | Intercept, verify identity, document |
| 5 | Infrastructure | Medium | Assess, report to maintenance, secure area |
| 6 | Traffic/Parking | Medium | Clear obstruction, issue warning/citation |
| 7 | Noise Complaint | Low | Verbal warning, document, follow up |
| 8 | Animal Control | Low | Contain if possible, contact animal services |
| 9 | General Complaint | Low | Document, route to appropriate department |

Supervisors and managers can override default priority when creating or triaging incidents.

### Incident Lifecycle

```
OPEN → ASSIGNED → IN_PROGRESS → [ESCALATED] → RESOLVED → CLOSED
  ↑                                                ↑        |
  └──── (reopened as new Open) ────────────────────┘────────┘

                          CANCELLED (from any state except Closed)
```

**States:**

| Status | Meaning | Who transitions |
|--------|---------|----------------|
| Open | Ticket created, no officer assigned | System (auto) or supervisor |
| Assigned | Officer notified, SLA response timer starts | System (smart dispatch) or supervisor |
| In Progress | Officer acknowledged or arrived on scene | Officer (mobile app) |
| Escalated | SLA breached or officer requested backup. Sub-flag `awaiting_external` pauses SLA timer when waiting on maintenance/external party. Timer resumes when flag is cleared by supervisor or external update. | System (auto) or officer/supervisor |
| Resolved | Issue addressed, pending supervisor review | Officer |
| Closed | Supervisor confirms resolution | Supervisor or manager |
| Cancelled | False alarm, duplicate, or test. Requires mandatory reason. | Supervisor or manager |

**Reopening:** A Resolved or Closed incident can be reopened by a supervisor. This creates a new incident linked via `related_incident_id` to the original, preserving the original's SLA history. The new incident starts fresh in Open status.

**Duplicates:** When a duplicate is identified, the newer incident is Cancelled with reason "duplicate" and linked to the original via `related_incident_id`.

### Incident Creation Sources

| Source | Flow |
|--------|------|
| Officer (mobile) | New Incident form → GPS auto-stamp → category picker → photo/note → submitted as Open |
| Supervisor (dashboard) | Incident form → manual location pin or zone select → priority override available |
| WhatsApp (resident) | Message received → AI triage → auto-categorized as Open → supervisor notified |
| Operations Room | Dashboard incident form → can create on behalf of phone caller or radio report |

### Smart Dispatch Flow

When a new incident is created:
1. **Deterministic scoring** ranks available officers:
   - Distance to incident (PostGIS, weighted 40%)
   - Current workload — active incidents assigned (weighted 30%)
   - Skill match — officer tags vs incident category (weighted 20%)
   - Time remaining in shift (weighted 10%)
2. **Top 3 candidates** presented to supervisor with scores
3. **Supervisor confirms** or selects a different officer
4. If no supervisor action within 2 minutes, **auto-assigns #1 candidate**
5. **AI generates explanation** for the recommendation (visible to supervisor)
6. **Fallback if AI/scoring fails:** incident appears in supervisor's queue for manual assignment

## SLA Framework

### SLA Targets

| Priority | Response Time | Resolution Time |
|----------|--------------|-----------------|
| Critical | 5 min | 1 hr |
| High | 5 min | 4 hr |
| Medium | 5 min | 24 hr |
| Low | 5 min | 48 hr |

**Response time** = time from Assigned → In Progress (officer acknowledges or arrives)
**Resolution time** = time from Open → Resolved

### SLA Timer Behavior

- **Assignment SLA:** 2 minutes from Open → Assigned (enforced by smart dispatch auto-assign). If smart dispatch fails, ops room is alerted for manual assignment.
- **Response timer** starts when incident enters **Assigned** status
- **Resolution timer** starts when incident enters **Open** status
- **Timer pause:** Only when incident is in **Escalated** status AND `awaiting_external` flag is set by supervisor (e.g., waiting on maintenance team, external contractor). Timer resumes when supervisor clears the flag or external party updates the ticket. Internal escalations (SLA breach, backup requested) do NOT pause the timer.
- Timer color in dashboard: Green (>50% remaining) → Yellow (25-50%) → Red (<25%) → Flashing Red (breached)
- Breached SLAs are never reset — they remain flagged for reporting even after resolution

### Escalation Rules

**Parallel escalation model:** On SLA breach, zone supervisor AND shift commander are notified simultaneously. If still unresolved after a second threshold, security manager is notified.

**Response SLA breach (officer didn't acknowledge in 5 min):**

| Time | Action |
|------|--------|
| T+5 min | Push notification + SMS to assigned officer (reminder) |
| T+7 min | Alert zone supervisor AND operations room simultaneously. Incident highlighted red in dashboard. |
| T+10 min | Auto-reassign to next best available officer. Original officer flagged for performance review. |
| T+15 min | Alert assistant manager. If Critical priority: alert security manager immediately. |

**Resolution SLA breach:**

| Time | Action |
|------|--------|
| 75% of resolution time elapsed | Warning notification to assigned officer + supervisor |
| 100% (SLA breached) | Alert zone supervisor AND operations room simultaneously. Incident status auto-set to Escalated. |
| 150% of resolution time | Alert assistant manager |
| 200% of resolution time | Alert security manager. Incident flagged as SLA failure in reports. |

**Critical incidents bypass:** For Critical priority, assistant manager is notified at creation (not just on breach). Security manager notified if not resolved within 30 minutes regardless of SLA status.

### Escalation Notification Channels

| Channel | Used for |
|---------|----------|
| Push notification (mobile) | Officers, supervisors |
| Socket.IO real-time (dashboard) | Operations room, supervisors, managers |
| SMS (via gateway) | Backup for officers with poor connectivity |
| WhatsApp (outbound) | Resident status updates |

### Zone Scoping

Supervisors see only their assigned zone(s). A supervisor can be assigned to multiple zones. Zone assignment is managed by assistant managers or HR admin.

Operations room operators see all zones (they need full visibility for dispatch) but cannot modify system configuration.

## Patrol Workflows

### Patrol Route Assignment

1. Supervisor or assistant manager defines patrol routes per zone (ordered list of checkpoints with expected dwell time)
2. Routes assigned to shifts — each shift has a set of required patrol routes
3. Officer receives patrol assignment on mobile app at shift start

### Patrol Execution

1. Officer starts patrol → app shows route map with checkpoint markers
2. At each checkpoint, GPS proximity confirmation (50m radius) auto-logs arrival
3. If officer cannot reach a checkpoint (blocked road, emergency), they can skip with a mandatory reason note
4. Patrol completion percentage tracked in real-time on dashboard
5. If patrol not started within 1 hour of shift start, supervisor alerted

### Patrol Anomaly Detection (AI)

- Officer stationary for >20 min during active patrol → alert supervisor
- Patrol route taking >150% of estimated duration → flag for review
- Checkpoint consistently skipped across multiple shifts → flag for route redesign
- Officer deviating significantly from route (>500m) → alert supervisor

## Performance Metrics

Tracked per officer, per zone, per shift:

| Metric | Calculation |
|--------|------------|
| Average response time | Mean time from Assigned → In Progress |
| SLA compliance rate | % of incidents resolved within SLA |
| Patrol completion rate | % of assigned checkpoints confirmed |
| Incidents handled | Count per shift/week/month |
| Escalation rate | % of incidents that reached Escalated status |
| No-show rate | % of scheduled shifts with NO_SHOW |
| Shift punctuality | Average minutes early/late for check-in |

Metrics displayed on supervisor dashboard (zone-scoped) and manager dashboard (global). Weekly and monthly aggregates included in AI-generated reports.
